The Facade Pattern — Extended Guide

The Facade pattern provides a unified interface to a set of interfaces in a subsystem. It defines a higher-level API that makes the subsystem easier to use without exposing internal complexity.

Key Ideas

  • Simplifies usage of a complex system by providing one entry point.
  • Reduces dependencies between client code and subsystem classes.
  • Introduces a higher-level API while leaving subsystems intact.
  • Often used in libraries, frameworks, and legacy integration.

Quick Checklist

  • ✔️ Identify subsystem complexity.
  • ✔️ Create a Facade class.
  • ✔️ Aggregate calls to subsystem classes.
  • ✔️ Expose only the essential API.

Idiomatic Implementations

Java

class SubsystemA { void opA() { System.out.println("A"); } }
class SubsystemB { void opB() { System.out.println("B"); } }

class Facade {
  private final SubsystemA a = new SubsystemA();
  private final SubsystemB b = new SubsystemB();
  public void operation() {
    a.opA();
    b.opB();
  }
}

C#

class Facade {
  private Audio audio = new Audio();
  private Video video = new Video();
  public void PlayMovie(string file) {
    audio.Load(file);
    video.Load(file);
  }
}

JavaScript

class CPU { start() { console.log("CPU start"); } }
class Memory { load() { console.log("Memory load"); } }

class ComputerFacade {
  constructor() {
    this.cpu = new CPU();
    this.memory = new Memory();
  }
  start() {
    this.cpu.start();
    this.memory.load();
  }
}

Python

class SubA:
    def op(self): print("SubA")

class SubB:
    def op(self): print("SubB")

class Facade:
    def __init__(self):
        self.a, self.b = SubA(), SubB()
    def operation(self):
        self.a.op()
        self.b.op()

Pros & Cons

✅ Benefits

  • Simplifies usage of complex subsystems.
  • Improves readability and usability.
  • Encourages loose coupling.
  • Good entry point for libraries/frameworks.

⚠️ Pitfalls

  • May hide necessary functionality.
  • Too many facades → extra indirection.
  • Not a replacement for poor design.

Alternatives & Comparisons

  • Adapter: when the interface must change shape.
  • Decorator: to extend functionality without hiding subsystem.
  • Mediator: coordinates objects, Facade just simplifies API.
  • Proxy: controls access, not simplify complexity.

Real-World Examples

  • Database Client — db.query() instead of sockets, protocols, parsers.
  • GUI Frameworks — high-level UI components wrapping complex low-level drawing code.
  • Payment Gateway SDK — one method call instead of cryptography, API calls, error handling.

Anti-Patterns & Misuses

  • ❌ Turning Facade into a "God Object" with too many responsibilities.
  • ❌ Hiding critical subsystem features, making debugging harder.
  • ❌ Using Facade instead of fixing bad subsystem design.
